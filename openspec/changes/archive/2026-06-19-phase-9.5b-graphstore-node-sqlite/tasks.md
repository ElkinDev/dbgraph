RESOLVED design decisions — apply MUST NOT re-litigate these:
- `StatementHandle` = `{ run(...args: unknown[]): { changes: number }; get(...args: unknown[]): unknown; all(...args: unknown[]): unknown[] }` — run/get/all are VARIADIC (the store mixes object binds `run({id, ...})` AND positional `run(id, id)` / `all(kind)` / `get(id)`).
- `WritableSqliteHandle` = `{ prepare(sql): StatementHandle; exec(sql): void; transaction<T>(fn: () => T): () => T; pragma(name: string): void; close(): void }`. `transaction` returns a CALLABLE (matches better-sqlite3 `.transaction()`); call sites invoke the returned fn SEPARATELY (`const upsert = handle.transaction(() => {...}); upsert();`). `pragma` is WRITE-ONLY, returns `void`.
- `betterSqliteHandle(db)` = thin pass-through (native `.prepare`/`.exec`/`.transaction`/`.pragma`/`.close`; native `RunResult.changes` passed through as `{ changes }`).
- `nodeSqliteHandle(db)` = duck-typed wrapper over `node:sqlite` `DatabaseSync` (typed `unknown`, NO unconditional import); `transaction` SYNTHESIZED: `exec('BEGIN')` → `try { const r = fn(); exec('COMMIT'); return r } catch (e) { exec('ROLLBACK'); throw e }`; normalize `StatementSync.run` return to `{ changes }` regardless of native field naming.
- `createSqliteGraphStore({ driver })`: `driver?: 'better-sqlite3' | 'node:sqlite'`, DEFAULT `'better-sqlite3'` (byte-identical); `'node:sqlite'` opt-in with the `isNodeSqliteAvailable()` version gate (explicit error, NO silent fallback). Existing callers (`open-connections.ts`/CLI) pass no `driver` → stay on the default.
- node:sqlite `:memory:` caveat: requesting `PRAGMA journal_mode = WAL` is a NO-OP on `:memory:` (forced `journal_mode=memory`) — NOT an error; parity tests assert ROW RESULTS, never the `journal_mode` echo. Confirm WAL + `foreign_keys = ON` + FTS5 work on node:sqlite (use a temp FILE-backed db for the one WAL-rollback test if `:memory:` cannot exercise WAL frames).

### Dependency bottlenecks

- **Batch 1 gates EVERYTHING and is the load-bearing no-CI safety net.** `WritableSqliteHandle` + `betterSqliteHandle` (1.1) MUST land AND all four files refactor to it (1.2–1.5) AND the round-trip golden stay byte-identical (1.6) before any node:sqlite code exists. A golden drift in 1.6 is a HARD STOP — the refactor leaked behavior. The variadic `StatementHandle.run/get/all` shape (1.1) is non-negotiable: the store mixes object binds `run({id, ...})` AND positional `run(id, id)` / `all(kind)` / `get(id)`, so a non-variadic signature breaks the refactor.
- **1.1 (handle interface) gates 1.2–1.5** (all four files type against it) and **2.1** (`nodeSqliteHandle` implements the same surface). The `transaction` CALLABLE contract (returns `() => T`, invoked separately) is load-bearing — `upsertGraph`/`deleteNodes`/`putSnapshot`/`runMigrations` ALL do `const fn = handle.transaction(...); fn();`. A non-callable return breaks every transactional write.
- **2.1 (`nodeSqliteHandle`) gates 2.2 (transaction commit/rollback), 2.3 (factory node path), 2.4 (parity), 2.5 (portability).** The synthesized `BEGIN/COMMIT/ROLLBACK` (2.1) MUST be correct under WAL + `foreign_keys = ON`; better-sqlite3 is the ORACLE — the rollback-on-throw test (2.2) is the load-bearing correctness check for the synthesis and MUST be GREEN before any parity run.
- **2.3 (driver selection + version gate) gates 2.4/2.5** (they construct a node:sqlite-backed store via the factory). The `isNodeSqliteAvailable()` gate (reused from `src/adapters/engines/sqlite/driver.ts`) MUST throw explicitly on Node < 22.5 — NO silent fallback to better-sqlite3, or the parity guarantee is unverifiable.
- **node:sqlite suites are runtime-gated** with `describe.skipIf(!isNodeSqliteAvailable())` — they run GREEN on Node 22.5+ (local) and SKIP cleanly below, so the better-sqlite3 default path (Batch 1) never depends on the node:sqlite path being exercised.
